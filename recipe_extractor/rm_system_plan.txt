Plan: NLP Recommendation System + DBDaemon API Integration
TL;DR
Integrate your trained CRF NLP model into DBDaemon via new D-Bus methods. User enters recipe in ElectronApp → DBDaemon calls NLP to extract ingredients → compares vs inventory → returns shopping list + updated inventory. Database receives three new schemas: FSS-Request, inventory_history (audit trail), and extended current_inventory with version tracking.

Steps
Phase 1: Database Schema Updates (depends on nothing - can start immediately)

Modify FSS_Inventory.db

Add inventory_history table for complete audit trail:
id, food_id, quantity_before, quantity_after, confidence_score, image_path, change_reason (FRT_DETECTION, USER_MANUAL, RECIPE_COMPARISON), changed_by (user/system), changed_at, created_at
Add version tracking to current_inventory: add version_id (auto-increment), last_change_reason, last_changed_by
Add index: idx_inventory_history_food_id for quick change lookups
Verify FSS_Request.db structure

Ensure request table has: id, recipe_name, food_id, quantity, unit, created_at, request_batch_id
Add request_batch_id to group ingredients from same recipe (allows undo/clear recipe)
Phase 2: NLP Pipeline Setup in recommend_system (parallel with Phase 1)

Create folder structure

Create recommend_system/models/ directory
Move fss_ner_crf_optimized.joblib to recommend_system/models/
Create NLP API wrapper module (recommend_system/RecipeAnalyzerAPI.py)

Wrap existing RecipeAnalyzerEngine from notebook
API methods needed:
init_engine(model_path, recipe_db_path) → loads CRF model + recipe database into memory
extract_ingredients(recipe_name: str) → List[Dict] → returns [{food_id, ingredient_name, quantity, unit}, ...]
get_available_recipes() → List[str] → returns list of indexed recipes (from training data)
Error handling: recipe not found, inference timeout (>10ms), model load failure
Create requirements.txt for NLP module

List: scikit-crfsuite, pyvi (Vietnamese tokenization), joblib dependencies
Phase 3: DBDaemon API Extensions (depends on Phase 2)

Update SqliteManager.py

Add methods:
insert_inventory_history(food_id, qty_before, qty_after, confidence, image_path, change_reason, changed_by) → logs to inventory_history
get_inventory_history(food_id, limit=50) → retrieves audit trail
insert_request_batch(recipe_name, ingredients_list) → inserts all ingredients for recipe, returns request_batch_id
get_all_requests() → modified to group by recipe_name, returns structured dict
clear_request_batch(request_batch_id) → clears old recipe from FSS-Request
Create new module db_daemon/src/RecommendationEngine.py

Orchestrates comparison logic:
compare_inventory_and_requests() → returns {available: [...], needed: [...], missing: [...]}
generate_shopping_list(recipe_name) → calls NLP → inserts to FSS-Request → calls comparison → returns formatted list
Error handling with ASPICE-compliant logging
Update DbDbusInterface.py

Add 4 new D-Bus methods:
CompareInventoryAndRequest() → returns JSON: {available: [{food_id, qty, image_path}], needed: [{food_id, qty}], missing: [{food_id, qty}]}
GenerateShoppingList(recipe_name: str) → calls NLP pipeline + comparison → returns same JSON format
GetCurrentInventory() → returns all items with version info and last change reason
UpdateInventoryFromNotification(food_id, qty, confidence_score, image_path, source) → updates inventory + logs to history (source = "FRT_DETECTION" or "USER_MANUAL")
Add D-Bus signal: RecommendationUpdated(recipe_name, shopping_list_json, inventory_json)
Update DbDaemonMain.py

Initialize RecommendationEngine on startup (after SqliteManager)
Load NLP model asynchronously to avoid blocking startup
Add NLP model health check on startup

Decisions
NLP runs in DBDaemon process (not separate service) for simplicity + lower latency
Complete audit trail (not sampled) for ASPICE compliance and potential regulatory requirement
Recipe database from training data (250 Vietnamese recipes) — no external API call dependency
FSS-Request table remains separate (not merged into current_inventory) to maintain recipe-ingredient mapping
No quantization loss — joblib compression sufficient for edge device (already <0.1MB)
3.22ms latency acceptable — no caching/pre-computation needed at Phase 1
Further Considerations
Recipe database expansion — Do you want to hardcode all 250 training recipes in a JSON file, or support dynamic recipe addition from external sources later?

Recommendation: Start with training data recipes in recommend_system/recipes.json, leave architecture open for DB-backed recipes
Quantity normalization — Your notebook defaults missing units to "1". Should DBDaemon support unit conversion (kg→g) for shopping list display?

Recommendation: Not in Phase 1 — add post-launch if needed
FRT source attribution — When FRT detects food and triggers UpdateInventoryFromNotification(), should it include which camera/sensor detected it?

Recommendation: Yes, add source_device field for multi-camera setups later
